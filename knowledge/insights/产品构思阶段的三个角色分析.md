
## 产品构思阶段的三个角色分析

### 1️⃣ `/office-hours` — YC 办公时间顾问

**角色定位：** YC（Y Combinator）办公时间合伙人

**做了什么事情：**

这个角色是整个流程的**起点**，它的核心任务是**在写任何代码之前，确保问题被真正理解**。它产出的是设计文档，不是代码。

具体工作流：

- **Phase 1：上下文收集** — 读取项目现有文件、git 历史、已有设计文档，然后通过一个关键问题判断用户的目标类型
- **根据目标分流到两种模式：**
  - **Startup 模式（Phase 2A）**：扮演严厉的 YC 合伙人，通过 **6 个逼问式问题** 诊断创业想法：
    1. **需求真实性** — "你有什么最强证据证明有人真正需要这个？"
    2. **现状分析** — "用户现在是怎么解决这个问题的？"
    3. **绝望的具体性** — "说出最需要这个产品的那个人的名字、职位"
    4. **最窄楔子** — "最小可付费版本是什么？"
    5. **观察与意外** — "你看过用户不借助帮助使用产品吗？"
    6. **未来适配性** — "3 年后世界变了，你的产品更重要还是更不重要？"
  - **Builder 模式（Phase 2B）**：扮演热情的协作者，帮用户找到想法的**最酷版本**
- **Phase 2.75：景观感知** — 搜索行业现有方案，运用三层知识体系（经典方法→当前趋势→第一性原理）
- **Phase 3：前提挑战** — 在提出任何方案前，质疑核心假设
- **Phase 4：方案生成** — 必须产出 2-3 个不同实现路径（最小可行版 + 理想架构 + 创意方案）
- **Phase 5：设计文档** — 输出完整设计文档到 `~/.gstack/projects/` 目录
- **Phase 6：交接** — 根据创始人信号强度，给出个性化反馈 + YC 申请推荐

**核心原则：** 反谄媚（禁止说"好问题！"）、逼问到具体（"企业"不是客户，"Sarah 在 Acme 公司"才是）、一次只问一个问题。

### 2️⃣ `/plan-ceo-review` — CEO 级计划审查员

**角色定位：** 不是来盖橡皮章的，而是来把计划变成**非凡**的

**做了什么事情：**

这个角色接手 `/office-hours` 产出的设计文档（或任何现有计划），从 CEO/创始人的战略高度进行**全面审查**。

具体工作流：

- **预审系统审计** — 读取代码库历史、现有文档、之前的设计文档和交接笔记
- **Step 0：核弹级范围挑战 + 模式选择**：
  - **0A 前提挑战** — 这是解决的正确问题吗？如果什么都不做会怎样？
  - **0B 现有代码杠杆** — 哪些已有代码能复用？是否在重复造轮子？
  - **0C 梦想状态映射** — 画出 `当前状态 → 这个计划 → 12 个月后的理想状态`
  - **0C-bis 实施替代方案（强制）** — 必须产出 2-3 个不同实现路径
  - **0D 模式特定分析** — 根据选中的模式进行深度分析
  - **0E 时间审讯** — 预判实现过程中每个小时会遇到什么决策
  - **0F 模式选择** — 4 种审查模式：
    1. **SCOPE EXPANSION**（范围扩展）— "这够大吗？找到 10 倍版本"
    2. **SELECTIVE EXPANSION**（选择性扩展）— "保持基线，但展示所有机会让我挑"
    3. **HOLD SCOPE**（保持范围）— "范围正确，让它防弹"
    4. **SCOPE REDUCTION**（范围缩减）— "砍到最核心"
- **10 个审查章节**：架构、错误处理、安全威胁、数据流、代码质量、测试、性能、可观测性、部署、长期轨迹，外加可选的 UI/UX 审查
- **18 种 CEO 认知模式**：包括分类本能（Bezos 单向门/双向门）、逆向思维（Munger）、聚焦即减法（Jobs 从 350 个产品砍到 10 个）等

**核心原则：** 零静默失败、每个错误都有名字、数据流有 4 条路径（正常/nil/空/错误）、图表是强制的。

### 3️⃣ `/autoplan` — 自动审查流水线

**角色定位：** 全自动审查编排器——"一条命令进，完整审查出"

**做了什么事情：**

这个角色是前两个角色的**自动化串联**。它不是独立的审查者，而是一个**编排器**，将三个审查技能（CEO → Design → Eng）**按严格顺序**自动执行，用 6 个决策原则代替人类判断。

具体工作流：

- **Phase 0：接收 + 存档** — 保存计划文件的恢复点，从磁盘加载三个审查技能的 SKILL.md 文件
- **Phase 1：CEO 审查** — 读取并执行 `/plan-ceo-review` 的全部内容，所有中间问题由 6 个原则自动回答，同时启动 Claude 子代理 + Codex 进行**双重独立评审**，产出 CEO 共识表
- **Phase 2：设计审查**（仅在检测到 UI 范围时）— 执行 `/plan-design-review`，同样双模型独立评审
- **Phase 3：工程审查** — 执行 `/plan-eng-review`，双模型独立评审
- **Phase 4：最终审批门** — 把所有自动决策和需要人类判断的"品味决策"汇总呈现给用户

**6 个自动决策原则：**

| # | 原则 | 含义 |
|---|------|------|
| 1 | **选择完整性** | 覆盖更多边界情况的方案优先 |
| 2 | **烧干湖水** | 爆炸半径内的所有问题一并修复 |
| 3 | **务实** | 两个方案效果一样，选更干净的 |
| 4 | **DRY** | 和现有功能重复？拒绝 |
| 5 | **显式优于聪明** | 10 行明显的代码 > 200 行抽象 |
| 6 | **倾向行动** | 合并 > 反复审查 > 停滞讨论 |

**决策分类：**
- **机械性决策** — 只有一个正确答案，静默自动处理
- **品味决策** — 合理的人会有分歧（接近的方案、边界范围、模型分歧），记录下来最后交给用户

### 三个角色的协作关系

```
/office-hours          /plan-ceo-review         /autoplan
  (发现问题)      →      (审查计划)           →   (全自动流水线)
  产出：设计文档          产出：完整审查报告         产出：三阶段审查 + 审批门
  
  ┌──────────┐     ┌────────────────┐     ┌───────────────────────┐
  │ 问对问题  │ ──→ │ 战略级审查     │ ──→ │ CEO→Design→Eng 全串联  │
  │ 写设计文档 │     │ 10个维度×4模式 │     │ 6原则自动决策          │
  │ 不写代码   │     │ 18种认知模式   │     │ 双模型独立评审          │
  └──────────┘     └────────────────┘     │ 品味决策交给人类        │
                                          └───────────────────────┘
```

**简单总结：**
- `/office-hours` 是**发现者** — "我们到底在解决什么问题？"
- `/plan-ceo-review` 是**审查者** — "这个计划够好吗？怎么让它变非凡？"
- `/autoplan` 是**编排器** — "让机器跑完全部审查，只把真正需要人类判断的留给你"
