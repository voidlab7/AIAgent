# gstack 设计阶段深度分析

## 一、设计阶段概览：从「好看」到「有灵魂」

设计阶段是 gstack 工作流中最体现**品味判断力**的环节。它的核心使命不是「让界面好看」，而是回答一个更深层的问题：

> **你的产品长什么样，以及为什么它应该长这个样子？**

三个角色形成了一条完整的设计生命线：

```
┌─────────────────────────────────────────────────────────────────┐
│                      设计阶段时间轴                               │
│                                                                 │
│  编码前（构建设计系统）    编码前（审查计划）    编码后（审计实现）  │
│  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐     │
│  │   design-    │ ──→ │ plan-design- │ ──→ │   design-    │     │
│  │ consultation │     │    review    │     │    review    │     │
│  └──────────────┘     └──────────────┘     └──────────────┘     │
│   产出: DESIGN.md       产出: 更好的计划      产出: 修复提交        │
│   (设计系统源文件)       (0-10 评分 × 7 维度)  (原子提交 + 报告)   │
└─────────────────────────────────────────────────────────────────┘
```

**设计三角的底层逻辑：**
- **design-consultation** = 定义标准（「我们的产品应该是什么感觉」）
- **plan-design-review** = 用标准审计计划（「你的计划能实现这种感觉吗」）
- **design-review** = 用标准审计实现（「你的代码实现了这种感觉吗」）

## 二、/design-consultation — 设计系统构建顾问

### 2.1 核心角色定位

> **"你是一个有强烈观点的高级产品设计师。你不展示菜单 —— 你倾听、思考、研究、提案。"**

这个角色最重要的设定是**顾问，不是表单向导**。传统的设计工具会给你一堆选项（"选一个颜色"、"选一种字体"），但 design-consultation 完全反过来：它主动提出一个**完整、自洽的设计方案**，然后邀请你调整。

### 2.2 六阶段工作流

```
Phase 0: 预检查
  ├── 检查 DESIGN.md 是否已存在
  ├── 从 README / package.json / 代码目录 推断产品上下文
  └── 读取 /office-hours 输出（如果有，产品上下文已预填充）
       ↓
Phase 1: 产品上下文理解
  ├── 一个问题覆盖所有需要知道的信息
  ├── 确认产品类型：Web App / Dashboard / 营销站 / 编辑器 / 内部工具
  └── 询问是否需要竞品研究
       ↓
Phase 2: 竞品研究（可选）
  ├── WebSearch 找 5-10 个同行产品
  ├── 浏览器截图 + 快照分析（如果可用）
  └── ★ 三层综合法 ★ → 发现 Eureka 洞察
       ↓
Phase 3: 完整提案 ← 灵魂阶段
  ├── 7 个维度一次性提出（美学 / 装饰 / 布局 / 颜色 / 字体 / 间距 / 动效）
  ├── ★ SAFE / RISK 分层 ★
  └── 一致性验证
       ↓
Phase 4: 细化调整（如果用户要求修改某部分）
       ↓
Phase 5: 字体 & 颜色预览页（HTML 可视化）
       ↓
Phase 6: 写入 DESIGN.md → 作为项目设计系统的 Source of Truth
```

### 2.3 三个核心设计思想

#### 思想一：三层竞品综合法（Three-Layer Synthesis）

这是 design-consultation 最精巧的分析框架：

```
Layer 1 — 已验证的共识（Tried and True）
│  "这个品类里所有产品共有的设计模式是什么？"
│   → 这些是桌面级标准，用户期望看到它们
│
Layer 2 — 新趋势（New and Popular）
│  "搜索结果和当前设计话语在说什么？什么在流行？"
│   → 不一定要跟风，但必须知道风在往哪吹
│
Layer 3 — 第一性原理（First Principles）
│  "基于我们对这个产品用户的了解——
│   传统的设计方案为什么可能是错的？
│   我们应该在哪里故意偏离品类规范？"
│   → 如果发现真正的洞察 → 标记为 EUREKA
```

为什么这个框架重要？因为大多数 AI 设计工具只做 Layer 1（复制行业标准），偶尔做 Layer 2（追随趋势）。但**真正好的设计**来自 Layer 3 —— 有理由地打破规则。

**Eureka 机制**是一个特别的亮点：

> "EUREKA: 每个 [品类] 产品都做 X，因为它们假设 [某个前提]。但这个产品的用户 [证据] —— 所以我们应该做 Y。"

这本质上是把**设计创新**变成了一个可追踪、可论证的推理过程，而不是拍脑袋。

#### 思想二：SAFE / RISK 分层决策

这是 Phase 3 完整提案的核心创新：

```
┌─────────────────────────────────────────────────┐
│                  完整设计提案                      │
│                                                  │
│  SAFE CHOICES（品类基线 —— 用户预期看到的）       │
│  ┌─────────────────────────────────────────┐    │
│  │ • 使用同品类标准的导航模式               │    │
│  │ • 遵循行业通用的颜色语义                 │    │
│  │ • 采用经过验证的信息架构                 │    │
│  └─────────────────────────────────────────┘    │
│                                                  │
│  RISKS（你的产品获得自己面孔的地方）              │
│  ┌─────────────────────────────────────────┐    │
│  │ • 一个品类内无人使用的字体选择 → 为什么  │    │
│  │ • 一个大胆的强调色 → 你得到什么 / 失去什么│    │
│  │ • 打破常规的布局方式 → 收益 / 代价       │    │
│  └─────────────────────────────────────────┘    │
│                                                  │
│  "安全选择让你在品类中识字。                      │
│   风险选择让你的产品被记住。"                     │
└─────────────────────────────────────────────────┘
```

这个框架的深层洞察是：**设计一致性只是入场券**。每个品类的产品都可以做到一致、专业，然后看起来一模一样。真正的问题是：**你在哪里冒险？** 设计师必须主动提出至少 2 个创意风险，并为每个风险解释为什么值得冒。

#### 思想三：一致性验证（Coherence Validation）

当用户修改了某一部分后，自动检查其余部分是否仍然协调：

```
用户操作: 将美学方向改为 "极简主义 / 粗野风"
                    ↓
自动检测:  你的动效设置是 "表现力丰富"
           粗野美学通常搭配极简动效
                    ↓
温和提醒:  "你的搭配不太常见 —— 如果是故意的没问题。
           要我建议匹配的动效方案吗？还是保持现在这样？"
                    ↓
尊重决定:  用户说保持 → 写入 DESIGN.md（永不拒绝执行）
```

**关键规则：提醒不一致，但永远不阻止。**

### 2.4 反 AI 烂俗（Anti-AI-Slop）体系

这是贯穿整个设计阶段的核心武器。gstack 定义了 **10 个 AI 生成界面的典型标志**：

| # | AI 烂俗模式 | 为什么是问题 |
|---|------------|-------------|
| 1 | 紫色/靛蓝渐变背景 | 几乎所有 AI 工具的默认配色 |
| 2 | **三列特性网格**（圆形图标 + 粗标题 + 两行描述 ×3） | 最典型的 AI 布局标志 |
| 3 | 彩色圆圈中的图标作为区域装饰 | SaaS 模板起手式 |
| 4 | 所有内容居中（`text-align: center` 一把梭） | 缺乏层次感 |
| 5 | 统一的超大圆角 | 缺乏层级区分 |
| 6 | 装饰性 blob、浮动圆圈、波浪 SVG 分割线 | 空洞内容的遮羞布 |
| 7 | Emoji 作为设计元素 | 业余感 |
| 8 | 卡片左边框彩色条 | 过度使用的视觉模式 |
| 9 | 模板化英雄文案 "Welcome to X"、"Unlock the power of..." | 零品牌辨识度 |
| 10 | 千篇一律的节奏（英雄 → 三特性 → 推荐 → 价格 → CTA） | 没有叙事节奏 |

**判断标准只有一个：一个受人尊敬的设计工作室的人类设计师，会发布这个吗？**

### 2.5 字体黑名单与过度使用名单

design-consultation 维护着两个严格的名单：

**绝不推荐（黑名单）：**
Papyrus、Comic Sans、Lobster、Impact、Jokerman、Bleeding Cowboys、Permanent Marker、Bradley Hand、Brush Script、Hobo、Trajan、Raleway、Clash Display、Courier New（用于正文）

**绝不作为主要字体推荐（过度使用）：**
Inter、Roboto、Arial、Helvetica、Open Sans、Lato、Montserrat、Poppins

> 这些字体本身不差，但因为每个 AI 工具都在推荐它们，使用这些字体 = 你的产品看起来像 AI 生成的。

### 2.6 最终产出：DESIGN.md

DESIGN.md 是整个设计阶段的**源真实文件**（Source of Truth）。下游所有角色都会读取它：

```
DESIGN.md 作为中枢的辐射关系：

  /design-consultation  ───写入──→  DESIGN.md  ←──校准── /plan-design-review
                                       ↑
                                       │
                           /design-review 读取并审计
                           /qa 在发现 UI 不匹配时标记
                           /review 在代码审查时检查是否偏离
                           所有后续 UI 开发都必须先读取
```

## 三、/plan-design-review — 设计维度 0-10 评分审查

### 3.1 核心角色定位

> **"你是一个审查计划的高级产品设计师 —— 不是审查线上网站。你的任务是在实现之前，找到缺失的设计决策并添加到计划中。"**

**关键区别：**
- design-consultation → 创建设计系统（DESIGN.md）
- plan-design-review → 用设计之眼审查实现计划
- design-review → 审计已实现的网站

plan-design-review 的产出是**更好的计划**，不是关于计划的文档。

### 3.2 九条设计原则

这是 plan-design-review 的底层信条体系，每一条都可以追溯到具体的设计大师或理论：

```
1. 空状态是功能        "没有条目。" 不是设计。每个空状态
                       需要温暖、主操作和上下文。

2. 每个屏幕有层次      用户先看什么？第二看什么？第三看什么？
                       如果所有东西都在争夺注意力，什么都赢不了。

3. 具体胜过氛围        "干净、现代的 UI" 不是设计决策。
                       命名字体、间距比例、交互模式。

4. 边界情况是用户体验   47 字符的名字、零结果、错误状态、
                       首次用户 vs 高级用户 —— 这些是功能，不是后续补丁。

5. AI 烂俗是敌人        如果看起来像其他所有 AI 生成的网站，就失败了。

6. 响应式 ≠ 手机上堆叠  每个视口需要有意图的设计。

7. 无障碍不是可选项      键盘导航、屏幕阅读器、对比度、触摸目标 ——
                        在计划中指定，否则它们不会存在。

8. 默认做减法           如果一个 UI 元素没有赢得它的像素，砍掉它。
                        功能臃肿杀产品比缺少功能还快。

9. 信任在像素级建立      每个界面决策要么建立用户信任，要么侵蚀它。
```

### 3.3 十二条认知模式

这不是检查清单，而是**设计师看世界的方式**：

```
 1. 看系统，不看屏幕        → 永远不孤立评估；考虑前因后果和崩溃情况
 2. 共情即模拟              → 不是"我为用户感到…"，而是跑心理模拟：
                              信号差、单手操作、老板在看、第一次 vs 第 1000 次
 3. 层次即服务              → 每个决策回答"用户应该先看什么？"
 4. 约束崇拜                → "如果只能展示 3 样东西，哪 3 样最重要？"
 5. 提问反射                → 第一本能是提问，不是给意见
 6. 边界情况偏执            → 47 字符名字？零结果？断网？色盲？RTL 语言？
 7. "我会注意到吗？"测试    → 隐形 = 完美。最高赞美是没注意到设计
 8. 有原则的品味            → "感觉不对"可追溯到一个被打破的原则
                              品味是可调试的，不是主观的
 9. 默认做减法              → Dieter Rams: "尽可能少的设计"
                              John Maeda: "减去显而易见的，加上有意义的"
10. 时间维度设计            → 5 秒（本能）、5 分钟（行为）、5 年（情感）
                              同时为三个维度设计 (Don Norman)
11. 为信任而设计            → 每个设计决策建立或侵蚀信任
                              陌生人共享住所需要像素级的安全意图 (Airbnb)
12. 故事板化旅程            → 在碰像素前，为用户完整情感弧线画故事板
                              "白雪公主"方法：每个时刻是有情绪的场景 (Joe Gebbia)
```

### 3.4 七遍审查评分法（The 0-10 Rating Method）

这是 plan-design-review 最核心的方法论 —— 每个设计维度独立评分，循环迭代到满意为止：

```
┌──────────────────────────────────────────────────────────┐
│                七遍审查流程（每遍独立评分）                 │
│                                                          │
│  Pass 1: 信息架构          "用户先看什么、第二看什么？"    │
│  Pass 2: 交互状态覆盖      "加载/空/错误/成功/部分状态？"  │
│  Pass 3: 用户旅程 & 情感弧  "每一步用户的情绪是什么？"      │
│  Pass 4: AI 烂俗风险       "描述够具体吗？还是通用模式？"   │
│  Pass 5: 设计系统对齐       "和 DESIGN.md 一致吗？"        │
│  Pass 6: 响应式 & 无障碍    "手机/平板/键盘导航/屏幕阅读器？"│
│  Pass 7: 未决设计决策       "哪些模糊地带会困扰实现者？"     │
└──────────────────────────────────────────────────────────┘
```

**每一遍的工作模式完全相同：**

```
① 评分:  "信息架构: 4/10"
② 差距:  "是 4 分因为计划没有定义内容层次。
          10 分的标准是每个屏幕都有清晰的主次层级。"
③ 修复:  直接编辑计划，补充缺失部分
④ 重评:  "现在 8/10 —— 仍然缺少移动端导航层次"
⑤ 询问:  如果有真正需要用户决策的设计选择 → AskUserQuestion
⑥ 再修:  重复直到 10 或用户说"够好了，继续"
```

**铁律：每遍之间暂停等待反馈，一个问题一个问题地问，绝不批量。**

### 3.5 优先级层次

当上下文空间有限（计划很大、Token 紧张）时的优先级：

```
Step 0（范围评估）> 交互状态覆盖 > AI 烂俗风险 > 信息架构 > 用户旅程 > 其他

理由：
- Step 0 必须先做（确定审查焦点）
- 交互状态是最容易遗漏、对用户体验伤害最大的部分
- AI 烂俗风险在 AI 时代最被低估
- 信息架构是一切的基础
```

### 3.6 UI 范围检测——智能退出

一个经常被忽略但非常重要的细节：

> **如果计划不涉及任何 UI 变更（纯后端改动），直接告诉用户"设计审查不适用"并退出。不要强行在后端变更上做设计审查。**

这体现了 gstack 的务实精神——工具只在真正需要时使用。

## 四、/design-review — 设计审计 + 修复循环

### 4.1 核心角色定位

> **"你是一个高级产品设计师兼前端工程师。用苛刻的视觉标准审查上线的网站 —— 然后修复你发现的问题。"**

**关键区别：** 前两个角色只看文档（DESIGN.md 和计划），design-review 看的是**真正渲染出来的网站**。它打开浏览器，截图，测量实际的字体大小、颜色值、间距，然后直接修代码。

### 4.2 多模式审计

```
┌──────────────────────────────────────────────────────────────┐
│                    审计模式选择                                 │
│                                                               │
│  Full（默认）   5-8 页，完整清单，响应式截图，交互流测试         │
│  Quick          首页 + 2 关键页，最快出分                       │
│  Deep           10-15 页，每个交互流，详尽清单，适合上线前审计   │
│  Diff-aware     功能分支自动触发 → 只审计分支影响的页面          │
│  Regression     对比历史基线 → 找到退化                         │
└──────────────────────────────────────────────────────────────┘
```

### 4.3 十一阶段审计流程

```
Phase 1:  第一印象            → 设计师直觉反应，不分析，先感受
Phase 2:  设计系统提取        → 从渲染结果反推实际使用的字体、颜色、间距
Phase 3:  逐页视觉审计        → 10 类约 80 项检查清单
Phase 4:  交互流审查          → 走 2-3 个关键用户流程，评估"感觉"
Phase 5:  跨页一致性          → 导航栏、页脚、组件是否跨页一致
Phase 6:  编译报告 + 双评分    → 设计分数 A-F + AI 烂俗分数 A-F
Phase 7:  分类排序            → 高/中/低影响 → 哪些修、哪些延后
Phase 8:  修复循环 ★          → 找代码 → 最小修复 → 原子提交 → 重新验证
Phase 9:  最终审计            → 修完后重新跑全部审计，确保分数没退化
Phase 10: 完整报告            → 含前后截图、提交 SHA、文件变更
Phase 11: TODOS.md 更新       → 延后的发现写入技术债务
```

### 4.4 Phase 1「第一印象」—— 设计师最独特的能力

这是 design-review 最与众不同的阶段。它不是检查清单，而是**设计师的直觉判断**：

```
"这个网站传达了 [什么]。"        → 一眼看到什么？专业？随意？混乱？
"我注意到 [观察]。"             → 什么突出？正面或负面——要具体
"我的眼睛依次看到：[1][2][3]。"  → 层次检查——这些是有意为之的吗？
"如果一个词形容：[词]。"         → 直觉判定
```

> "设计师不犹豫 —— 他们做出反应。"

### 4.5 80 项审计清单（10 大类）

design-review 拥有 gstack 中最详尽的设计审计清单：

```
类别                    项目数    权重    审查焦点
─────────────────────  ────── ────── ──────────────
1. 视觉层次 & 构图       8     15%    焦点、眼动线、信息密度
2. 字体排印              15    15%    字体数量、行高、层级、黑名单
3. 颜色 & 对比           10    10%    WCAG AA、语义色、暗模式
4. 间距 & 布局           12    15%    网格一致性、4/8px 基准、响应式
5. 交互状态              10    10%    悬停/聚焦/加载/空/错误
6. 响应式设计             8    10%    移动端不是堆叠、触摸目标 44px
7. 动效 & 动画            6     5%    缓动函数、prefers-reduced-motion
8. 内容 & 微文案           8    10%    空状态温暖、错误信息具体、按钮标签
9. AI 烂俗检测           10     5%    10 大反模式（独立评分 + 综合权重）
10. 性能即设计             6     5%    LCP、CLS、骨架屏、字体加载
```

### 4.6 Phase 8「修复循环」—— 最体现工程纪律的环节

修复循环是 design-review 的执行引擎，每一步都有严格纪律：

```
8a. 定位源码        → 搜索 CSS 类名、组件名、样式文件
8b. 修复            → 最小修复，CSS 优先于结构变更
8c. 原子提交        → 一个修复一个提交: "style(design): FINDING-NNN — 描述"
8d. 重新验证        → 回到页面截图对比
8e. 分类            → verified / best-effort / reverted
8e.5 回归测试       → CSS-only 跳过；涉及 JS 行为的写回归测试
8f. 自我调节        → 每 5 个修复评估风险
```

**自我调节机制（风险控制）：**

```
DESIGN-FIX 风险计算:
  起始:                              0%
  每次回退(revert):                 +15%
  每个 CSS-only 文件变更:            +0%   ← 安全：纯样式
  每个 JSX/TSX/组件文件变更:         +5%   每个文件
  修复 10 个之后:                    +1%   每多一个
  触碰不相关文件:                   +20%

  > 20% → 立刻停止，询问用户
  硬限制: 30 个修复
```

这个机制的智慧在于：它把 CSS-only 变更视为安全（可逆、影响小），但对组件级变更保持警惕。回退（revert）给了最重的惩罚，因为回退意味着修复引入了回归。

### 4.7 双重评分体系

design-review 的最终报告有两个独立的标题分数：

```
┌────────────────────────────────────┐
│  Design Score: B                    │  ← 10 类加权平均
│  AI Slop Score: D                   │  ← 独立评分 + 一句话判定
├────────────────────────────────────┤
│  评分标准:                          │
│  A = 有意图、精致、令人愉悦          │
│  B = 基础扎实、小瑕疵               │
│  C = 能用但通用、没有设计观点        │
│  D = 明显问题、感觉未完成           │
│  F = 主动伤害用户体验               │
└────────────────────────────────────┘
```

AI Slop Score 独立存在的原因：大多数开发者无法判断自己的网站是否看起来像 AI 生成的，但设计师可以。这是 design-review 最独特的超能力。

           │             │             │
     写入(创建)      读取(校准)      读取(审计)
           │             │             │
  design-consultation  plan-design    design-review
           │           review           │
           │             │             │
     从产品上下文    从实现计划      从渲染结果
     和竞品研究      和设计原则      和源代码
     构建设计系统    审查设计完备性   发现并修复偏差
```

### 5.2 共享的设计硬规则（Design Hard Rules）

三个角色共享一套**页面类型分类器 + 对应规则**：

```
第一步：分类页面类型
  ├── 营销/落地页（英雄驱动、品牌导向、转化为核心）
  ├── 应用 UI（工作区驱动、数据密集、任务导向）
  └── 混合（营销外壳 + 应用功能区域）

第二步：应用对应规则集

  营销页规则:
  - 首屏作为一个构图整体，不是仪表盘
  - 品牌优先层次：品牌 > 标题 > 正文 > CTA
  - 英雄必须全出血（full-bleed），不能有内嵌/圆角变体
  - 英雄预算：品牌 + 一个标题 + 一句支撑 + 一组 CTA + 一张图
  - 动效：至少 2-3 个有意图的动画
  - 美丽的默认值：构图优先、品牌最大字、最多两种字体、无卡片

  应用 UI 规则:
  - 平静的表面层次、强字体排印、少量颜色
  - 密集但可读、最少装饰
  - 组织方式：主工作区 + 导航 + 次要上下文 + 一个强调色
  - 避免：仪表盘卡片马赛克、粗边框、装饰渐变
  - 文案：工具语言 —— 定位、状态、操作

  通用规则:
  - CSS 变量定义颜色系统
  - 不用默认字体栈
  - 每个区域只做一件事
  - 卡片必须赢得存在权 —— 禁止装饰性卡片网格
```

### 5.3 交叉模型验证（Outside Voices）

三个角色都支持「外部设计声音」机制 —— 调用外部 AI 模型进行平行审查，获得独立视角：

```
┌─────────────────────────────────────────────────────────────┐
│                 外部设计声音（Outside Voices）                 │
│                                                              │
│  Claude 主审查  ←──对比──→  Codex 独立审查                    │
│       │                         │                            │
│  各自独立评估 7 项 Litmus Check                               │
│       │                         │                            │
│       └────────┬────────────────┘                            │
│                ↓                                             │
│         共识评分表                                            │
│   CONFIRMED = 两者一致                                       │
│   DISAGREE  = 两者分歧 → 需要人类决策                        │
│   NOT SPEC'D = 信息不足                                      │
└─────────────────────────────────────────────────────────────┘
```

每个角色的 Outside Voices 提示语是不同的（根据角色职责定制）：

| 角色 | Codex 提示焦点 | Claude 子代理提示焦点 |
|------|---------------|---------------------|
| design-consultation | 视觉论点、内容计划、交互创意 | 惊喜的设计方向 |
| plan-design-review | 硬拒绝 + Litmus Check + 硬规则 | 信息层次、缺失状态、用户旅程、具体性 |
| design-review | 源码级审查（间距系统、颜色系统、响应式） | 跨文件一致性模式 |

### 5.4 设计批评格式

三个角色共享一种结构化反馈语言：

```
"我注意到..."  → 观察（"主 CTA 和次级操作在争夺注意力"）
"我好奇..."    → 提问（"用户会理解'处理'在这里的意思吗？"）
"如果..."      → 建议（"如果把搜索移到更显眼的位置呢？"）
"我认为...因为..."  → 有理由的意见（"间距太均匀因为没有创建层次"）
```

**一切都要关联到用户目标和产品目标。问题旁边永远要有具体的改进建议。**

## 六、核心设计哲学总结

### 6.1 三个角色本质上在问三个不同层次的问题

```
design-consultation:  "我们的产品应该给用户什么感觉？"
                       → 定义设计意图

plan-design-review:   "我们的计划能实现这种感觉吗？"
                       → 验证设计完备性

design-review:        "我们的实现真的实现了这种感觉吗？"
                       → 验证设计执行力
```

### 6.2 贯穿始终的五个核心理念

1. **顾问，不是菜单**（Consultant, not wizard）
   - 主动提出有观点的方案，而不是列出 100 个选项让用户选
   - 每个推荐都必须有理由（"因为 Y"，而不是"我推荐 X"）

2. **品味是可调试的**（Taste is debuggable）
   - "感觉不对"永远可以追溯到一个被打破的原则
   - 不是主观的，而是有原则支撑的审美判断

3. **一致性是入场券，风险才是灵魂**（Coherence is table stakes, risk is soul）
   - 设计系统的一致性只是基础
   - 真正让产品被记住的是有理由的创意风险

4. **AI 烂俗是当代设计的首要敌人**（AI slop is the enemy）
   - 10 个反模式清单贯穿所有角色
   - 独立的 AI Slop Score 评分
   - 硬拒绝准则 + Litmus Check 交叉验证

5. **永远不阻止，只提醒**（Nudge, never block）
   - 提醒一致性问题，但尊重用户最终决定
   - 永远接受用户的选择，永远不拒绝写入 DESIGN.md

### 6.3 与其他阶段的衔接

```

                                               │
 /office-hours 产出      design-consultation    /plan-eng-review
 的产品上下文      ──→   直接读取并预填充        读取 DESIGN.md

## 七、为什么说这是 gstack 最体现「品味」的阶段

传统的设计工具和 AI 设计助手通常做的事情是：给你颜色选择器、字体浏览器、模板库。它们把设计变成了「选配置」。

gstack 的设计阶段做了一件根本不同的事：**它把设计变成了可追踪的推理过程。**

- 竞品研究的三层综合法，让「创新」变得可论证
- SAFE/RISK 分层，让「创意风险」变得可管理
- 0-10 评分法，让「好不好看」变得可度量
- AI 烂俗清单，让「是否独特」变得可检测
- 一致性验证，让「协调性」变得可自动化
- 修复循环的风险控制，让「改进」变得安全可控

**最终，这个阶段要回答的不是「用什么颜色」，而是「为什么用这个颜色，以及这个选择如何服务于你的产品和用户」。**
