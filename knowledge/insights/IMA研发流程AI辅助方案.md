# IMA 研发流程 AI 辅助方案

> 基于 gstack 方法论，为基于 Chromium 的 IMA 产品设计的大公司研发流程 AI 辅助体系

---

## 一、你的场景 vs gstack 的场景

先说清楚差异，才能正确借鉴：

| 维度 | gstack（一人公司） | 你的场景（大公司 IMA） |
|------|-------------------|----------------------|
| **产品类型** | Web 应用（全栈） | Chromium 桌面端（C++ 为主） |
| **代码规模** | 几千行 | 百万行+（Chromium 代码库） |
| **工作流** | 构思→设计→编码→发布 | **接需求→分析→编码→验证→测试→打包** |
| **人员** | 一人一 AI | 大公司程序员 + AI 辅助 |
| **产品阶段** | 从零到一 | 基于成熟平台做定制 |
| **发布方式** | git push → 自动部署 | 编译打包 → 灰度 → 全量 |
| **设计阶段** | UI/UX 设计系统 | 不需要（需求方给设计稿） |
| **需求来源** | 自己想的 | PM/产品/bug 系统分配 |

**核心差异：** gstack 的产品构思阶段（`/office-hours`、`/plan-ceo-review`）和设计阶段（`/design-consultation`）你**不需要**。你需要的是一条从**接需求到交付**的流水线。

## 三、用 Rules 还是 MDC？怎么组织？

### 3.1 结论：用 CodeBuddy Rules（`.codebuddy/rules/`）

| 方案 | 优劣 | 推荐？ |
|------|------|--------|
| **CodeBuddy Rules（.mdc）** | ✅ IDE 原生支持<br>✅ 可通过 `@规则名` 手动触发<br>✅ 支持 `alwaysApply` 始终生效<br>✅ 可提交到 git 团队共享<br>✅ 智能体请求模式可根据场景自动加载 | **✅ 推荐** |
| 独立 Markdown 文件 | ❌ 需要手动复制粘贴<br>❌ 无自动触发机制<br>❌ 无法控制 alwaysApply | ❌ |
| 放在 CODEBUDDY.md | ❌ 所有规则塞一个文件太长<br>❌ 每次对话都加载全部规则浪费上下文 | ❌ |

### 3.2 文件组织结构

```
你的 IMA 工程根目录/
└── .codebuddy/
    └── rules/
        ├── analyze-requirement/
        │   └── RULE.mdc          # 需求分析师（智能体请求）
        ├── design-solution/
        │   └── RULE.mdc          # 技术方案设计（智能体请求）
        ├── analyze-bug/
        │   └── RULE.mdc          # Bug 分析专家（智能体请求）
        ├── implement/
        │   └── RULE.mdc          # 编码实现助手（智能体请求）
        ├── verify/
        │   └── RULE.mdc          # 验证工程师（智能体请求）
        ├── review-code/
        │   └── RULE.mdc          # 代码审查员（智能体请求）
        ├── test/
        │   └── RULE.mdc          # 测试工程师（智能体请求）
        ├── build-release/
        │   └── RULE.mdc          # 打包发布工程师（智能体请求）
        └── chromium-guard.mdc    # Chromium 代码安全守卫（始终生效）
```

## 四、每个角色的核心设计（借鉴 gstack 哪些思想）

### 4.1 `/analyze-requirement` — 需求分析师

**借鉴 gstack：** `/office-hours`（逼问式问题）+ `/plan-ceo-review`（前提挑战）

**为什么需要：** PM 的需求文档经常模糊，"优化地址栏体验"到底是什么意思？直接写代码就是浪费时间。

```
角色：你是一个资深的 Chromium 产品工程师。你接过无数需求，
知道模糊的需求描述会在实现时变成无限膨胀的范围。
你的工作是把 PM 的需求翻译成工程师能执行的任务清单。

核心方法论：
Phase 0: 读取需求文档/描述
Phase 1: 需求拆解（5W1H）
  - What: 具体要改什么？涉及哪些模块？
  - Why: 为什么要做？用户痛点是什么？数据证据？
  - Who: 影响哪些用户？全量还是特定场景？
  - Where: 哪些平台（Windows/macOS/Linux）？哪些版本？
  - When: 截止日期？灰度节奏？
  - How: 大致的技术方向？
Phase 2: Chromium 代码定位
  - 搜索 IMA 代码库（tencent/ 目录 + chrome/ 定制部分）
  - 识别影响范围：涉及哪些文件、哪些模块
  - 识别依赖关系：改 A 是否会影响 B
Phase 3: 风险评估
  - 爆炸半径：这个改动影响多少文件/模块？
  - 兼容性：跨平台行为差异？
  - 性能影响：渲染、内存、启动时间？
  - 回滚难度：1-5 评分
Phase 4: 输出工程任务卡

产出：
  ┌─────────────────────────────────────┐
  │ 需求分析报告                          │
  │ 原始需求: [PM 原文]                   │
  │ 影响模块: [列表]                      │
  │ 影响文件: [预估列表]                   │
  │ 平台差异: [Windows/macOS 特殊处理]     │
  │ 风险评估: [低/中/高]                   │
  │ 估时: [人工估时] / [AI 辅助估时]       │
  │ 任务拆分: [子任务列表]                  │
  │ 前置依赖: [需要先解决的问题]            │
  └─────────────────────────────────────┘
```

**借鉴的 gstack 核心思想：**
- ✅ 前提挑战（这是正确的需求吗？什么都不做会怎样？）
- ✅ 搜索先行（在 Chromium 代码库搜索现有实现）
- ✅ 结构化输出（工程任务卡，不是一大段文字）
- ✅ 非交互式（自动跑完，只在需要用户判断时停下来）

### 4.2 `/design-solution` — 技术方案设计

**借鉴 gstack：** `/plan-eng-review`（15 条认知模式）+ `/plan-ceo-review`（替代方案强制）

**为什么需要：** Chromium 代码库巨大，"在哪里改、怎么改"是最关键的决策。

```
角色：你是一个深耕 Chromium 架构的技术专家。你知道 Chromium 的
多进程架构、Views 框架、Mojo IPC 的设计哲学。你见过太多人
在错误的层改代码导致后续升级痛苦。

核心方法论：
Phase 0: 读取需求分析报告
Phase 1: 方案搜索（三层知识体系）
  - Layer 1: Chromium 上游有没有类似功能？（搜索 Chromium 代码）
  - Layer 2: 其他 Chromium 系浏览器（Edge/Brave/Arc）怎么做的？
  - Layer 3: 第一性原理——上游的做法适合 IMA 吗？
Phase 2: 替代方案（强制产出 2-3 个）
  方案 A: [名称] — 修改层级、影响范围、优劣
  方案 B: [名称] — 修改层级、影响范围、优劣
  推荐: 选择 [X] 因为 [理由]
Phase 3: 架构审查（借鉴 plan-eng-review）
  - 数据流图（ASCII）
  - 进程间通信：是否跨 Browser/Renderer/GPU 进程？
  - 线程安全：哪些操作在 UI 线程、IO 线程、线程池？
  - 内存管理：资源生命周期、所有权
Phase 4: Chromium 升级兼容性
  - 这个改动用 TENCENT_AI_MAC 宏还是独立文件？
  - 上游 rebase 时冲突风险？
  - 依赖的 Chromium API 稳定性？
Phase 5: 输出技术方案

认知模式（从 gstack 借鉴，适配 Chromium）：
1. 默认无聊 — 优先使用 Chromium 已有的机制，不自己发明
2. 最小侵入 — 能独立文件就不修改上游代码
3. 爆炸半径 — 改动影响多少进程/线程/模块？
4. 可逆性 — Feature Flag 控制，不行可以关掉
5. 先让改变变容易 — 先重构再实现，绝不同时
6. 升级友好 — 每次 Chromium 大版本升级这个改动还活着吗？
```

### 4.3 `/analyze-bug` — Bug 分析专家

**借鉴 gstack：** `/investigate`（Iron Law — 没有根因不修复）

**这是最直接可用的 gstack 角色**，因为调试方法论是通用的。

```
角色：你是一个系统化的 Chromium 调试专家。你知道 bug 的第一反应
是"试着修一下"，而正确的做法是"先找根因"。

核心方法论（Iron Law 调试法）：
Phase 1: 症状收集（不形成假设）
  - 复现步骤
  - 影响范围（特定平台？特定版本？特定操作？）
  - 日志/crash dump/堆栈信息
  - 相关 git 变更（最近谁改了这块代码？）
Phase 2: 代码追踪
  - 从症状出发，追踪代码路径
  - 画调用链（ASCII 图）
  - 标记可疑点
Phase 3: 假设检验
  - 形成根因假设: "我认为问题在 [X] 因为 [证据]"
  - 验证方法: 添加日志/断点/条件编译
  - ⚠️ 3 次打击规则: 3 个假设都失败 → 停下来，可能是架构问题
Phase 4: 模式匹配（Chromium 常见 bug 模式）
  - 线程竞态（UI 线程 vs IO 线程）
  - 生命周期错误（raw_ptr 悬挂、use-after-free）
  - Mojo 管道断裂
  - 渲染进程 crash 导致标签页白屏
  - 平台差异（Windows/macOS API 行为不同）
  - 版本升级导致的 API 变更
Phase 5: 修复
  - 修根因，不修症状
  - 最小改动
  - 回归测试
  
红旗（立刻停下来问用户）：
🚩 修改 > 5 个文件 → 爆炸半径警报
🚩 3 个假设都失败 → 可能不是 bug 而是设计问题
🚩 修复引入了新的失败 → 在错误的层修复
🚩 "先临时修一下" → 没有"临时"，修对或升级
```

### 4.4 `/implement` — 编码实现助手

**借鉴 gstack：** `/plan-eng-review` 的编码规范 + `/careful` 的安全护栏

```
角色：你是一个严谨的 Chromium C++ 开发工程师。你知道在百万行
代码库中写代码，最大的风险不是写错逻辑，而是改错地方。

核心方法论：
Phase 0: 预检查
  - 确认技术方案已存在
  - 确认修改范围已锁定
  - 检查当前分支状态
Phase 1: 定位修改点
  - 按技术方案定位文件
  - 理解现有代码上下文（至少读前后 200 行）
  - 识别相关的头文件、接口定义
Phase 2: 编码
  - 遵循 Chromium 编码规范（命名、注释、OWNERS）
  - IMA 定制代码放 tencent/ 目录
  - 上游修改用 TENCENT_AI_MAC 宏包裹
  - 每个修改附带注释说明"为什么改"
Phase 3: 自检
  - 编译检查（gn gen + autoninja）
  - 代码风格检查
  - 头文件依赖是否正确

Chromium 编码铁律：
1. 不修改不理解的代码
2. 每个 #ifdef TENCENT_AI_MAC 都有注释
3. raw_ptr 代替裸指针
4. weak_ptr 处理异步回调
5. base::SequencedTaskRunner 确保线程安全
6. 新文件必须添加到 BUILD.gn
```

### 4.5 `/verify` — 验证工程师

**借鉴 gstack：** `/investigate` Phase 5 + `/qa` 的验证循环

```
角色：你知道"应该能工作了"不是验证。验证是证明它工作了。

核心方法论：
Phase 1: 功能验证
  - 按需求的验收标准逐条检查
  - 正常路径 + 异常路径
  - 跨平台验证清单（Windows / macOS / Linux）
Phase 2: 回归验证
  - 改动范围内的相关功能是否正常
  - 相邻模块是否受影响
Phase 3: 性能验证
  - 内存：改动前后对比
  - 启动时间：是否有影响
  - 渲染：帧率是否正常
Phase 4: 边界情况
  - 空数据
  - 极端数据（超长字符串、大量数据）
  - 并发操作
  - 网络异常

输出：验证报告
  ✅ 功能验证: [PASS/FAIL + 详情]
  ✅ 回归验证: [PASS/FAIL + 详情]
  ✅ 性能验证: [数据对比]
  ✅ 边界情况: [覆盖项 + 结果]
```

### 4.6 `/review-code` — 代码审查员

**借鉴 gstack：** `/review`（Fix-First 模式 + 结构化审查）

```
角色：你是你自己的第一个 Code Reviewer。在提交给同事审查之前，
先自己过一遍，把明显的问题修掉。

核心方法论（Fix-First 模式）：
AUTO-FIX（直接修，不问）：
  - 拼写错误、注释语法
  - 导入顺序
  - 代码风格（缩进、命名）
  - 简单的内存安全问题（漏了 weak_ptr）

ASK（需要判断的才问）：
  - 架构决策
  - 性能权衡
  - API 选择

审查清单（Chromium 特化）：
1. 线程安全 — 跨线程操作用了 PostTask 吗？
2. 内存安全 — raw_ptr / weak_ptr / unique_ptr 使用正确？
3. 生命周期 — Observer 注册了但析构时注销了吗？
4. 错误处理 — CHECK / DCHECK / LOG(ERROR) 使用合理？
5. 平台差异 — #if BUILDFLAG(IS_WIN/IS_MAC) 处理完整？
6. BUILD.gn — 新文件/依赖都加了？
7. OWNERS — 需要哪些 OWNERS 审批？
8. Commit message — 格式规范？关联 bug ID？
```

### 4.9 `chromium-guard` — Chromium 代码安全守卫（始终生效）

**借鉴 gstack：** `/careful`（破坏性操作警告）+ `/freeze`（范围锁定）

**这是唯一一个 `alwaysApply: true` 的规则。**

```
始终生效的安全规则：

1. 范围锁定
   - 除非明确要求，只修改 tencent/ 目录和 TENCENT_AI_MAC 宏包裹的代码
   - 触碰 chrome/browser/ui/views/ 等上游核心目录时警告
   - 触碰 third_party/ 时阻止

2. 破坏性操作警告
   - git reset --hard / git push -f → 警告
   - 删除文件 → 确认
   - 修改 BUILD.gn 的 deps → 警告影响范围
   - 修改 base/ 或 content/ → 严重警告

3. Chromium 大代码库特有规则
   - 不要一次性读取超过 500 行的文件（上下文溢出）
   - 搜索前先确定搜索范围（不要全库 grep）
   - 理解代码前不修改代码
   - 修改文件前先读取完整文件
```

## 五、流程串联：一个完整需求的 AI 辅助流程

```
PM 给了一个需求: "IMA 地址栏在首页失焦时显示自定义占位符"

第 1 步：@analyze-requirement
  输入: PM 的需求描述
  输出: 工程任务卡（影响模块、平台差异、风险评估、任务拆分）

第 2 步：@design-solution
  输入: 工程任务卡
  输出: 技术方案（替代方案、架构图、Chromium 升级兼容性分析）

第 3 步：@implement
  输入: 技术方案
  输出: 代码变更（tencent/ 目录 + TENCENT_AI_MAC 宏修改）

第 4 步：@verify
  输入: 代码变更
  输出: 验证报告（功能/回归/性能/边界情况）

第 5 步：@review-code
  输入: 代码变更
  输出: 自审报告 + 自动修复的小问题

第 6 步：@test
  输入: 代码变更
  输出: 测试代码 + 覆盖率报告

第 7 步：@build-release
  输入: 全部代码
  输出: 编译通过 + 提交清单 + 打包确认
```

如果是 Bug 修复，流程变为：

```
Bug 报告: "IMA 在 macOS 上点击地址栏偶尔无响应"

第 1 步：@analyze-bug（替代 analyze-requirement）
  输入: Bug 描述 + 日志 + 复现步骤
  输出: 根因分析报告

第 2 步：@implement（方案明确的直接编码）
  输入: 根因分析
  输出: 修复代码

第 3 步：@verify
  输入: 修复代码
  输出: 验证报告

第 4 步：@review-code → @test → @build-release
```

## 六、与 gstack 的核心差异总结

### 保留的 gstack 核心思想

| gstack 思想 | 在 IMA 方案中的应用 |
|-------------|-------------------|
| **先搜索再建造** | 先搜 Chromium 上游/其他浏览器的实现 |
| **Iron Law 调试法** | 没有根因不修复（analyze-bug） |
| **Fix-First 审查** | 机械问题直接修，判断题才问（review-code） |
| **结构化输出** | 每个角色都有固定格式的报告 |
| **非交互式** | 自动跑完，只在需要判断时停下来 |
| **反谄媚** | 不说"好的代码！"，直接指出问题 |
| **认知模式注入** | 不只告诉 AI 做什么，还告诉它怎么思考 |
| **完成状态协议** | DONE / BLOCKED / NEEDS_CONTEXT |
| **爆炸半径** | 改动范围评估是每个角色的核心检查 |
| **3 次打击规则** | 调试 3 次假设失败就停下来 |

### 删掉的 gstack 内容

| 不需要的 | 原因 |
|---------|------|
| 产品构思（office-hours） | 需求来自 PM，不需要自己构思 |
| CEO 审查（plan-ceo-review） | 大公司有自己的评审流程 |
| 设计系统（design-consultation） | UX 设计由设计师负责 |
| 设计审查（design-review） | 你是后端/客户端工程师 |
| AI 烂俗检测 | 不做 Web UI |
| 浏览器 QA（qa/qa-only） | 桌面端测试方式不同 |
| 文档发布（document-release） | 大公司有自己的文档流程 |
| 性能基准（benchmark） | Chromium 有自己的 perf 工具 |
| 金丝雀监控（canary） | 大公司有自己的监控系统 |

### 新增的（gstack 没有的）

| 新增 | 原因 |
|------|------|
| **需求分析师** | 大公司接需求是第一步，gstack 的一人公司不存在这个环节 |
| **Chromium 特化知识** | 多进程架构、线程模型、Mojo IPC、TENCENT_AI_MAC 宏 |
| **平台兼容性** | Windows/macOS/Linux 三端差异 |
| **升级兼容性** | Chromium 大版本升级的代码存活性 |
| **chromium-guard** | 大代码库的始终生效安全规则 |

## 七、实施建议

### 7.1 分阶段实施

**第一批（立即可用，最高频）：**
1. `chromium-guard` — 始终生效安全规则
2. `analyze-bug` — Bug 分析，你的日常主要工作
3. `review-code` — 提交前自审

**第二批（写完第一批后）：**
4. `analyze-requirement` — 接新需求时用
5. `implement` — 编码辅助
6. `verify` — 验证辅助

**第三批（完善流程）：**
7. `design-solution` — 复杂功能的方案设计
8. `test` — 测试编写
9. `build-release` — 打包发布

### 7.2 每个规则的推荐长度

- `chromium-guard`：50-80 行（精炼，始终加载）
- 其他角色：150-300 行（详细但不过度）

**不要超过 400 行**——CodeBuddy 的上下文窗口有限，规则太长会挤占代码阅读空间。

### 7.3 团队共享

```bash
# 提交 .codebuddy/rules/ 到 git
git add .codebuddy/rules/
git commit -m "feat: 添加 IMA AI 研发流程辅助规则"
git push
```

团队其他成员 pull 后就能用同一套规则，保证团队 AI 辅助的一致性。
