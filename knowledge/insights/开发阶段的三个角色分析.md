# 开发阶段方法论深度分析

## 整体设计哲学

开发阶段的三个角色构成一个 **"锁定→调试→审计"三角**，分别守护代码的三个生命阶段：

```
              ┌─────────────────┐
              │  plan-eng-review │  ← 编码之前：锁定架构
              │    "工程经理"      │
              └────────┬────────┘
                       │
            代码写完之后 ↓ 出了 bug
                       │
              ┌────────┴────────┐
              │   investigate    │  ← 出问题时：系统化调试
              │   "调试专家"      │
              └────────┬────────┘
                       │
            修好之后 / ↓ 上线之前
                       │
              ┌────────┴────────┐
              │      cso        │  ← 全生命周期：安全审计
              │  "首席安全官"     │
              └─────────────────┘
```

三个角色有一个**共同的硬性纪律**：**先诊断，后行动。绝不跳过分析直接动手。**

---

## 一、`/plan-eng-review` — 工程经理级计划审查

### 核心方法论：**"写代码之前把设计锁死"**

**铁律：审查完成之前不写代码。**

这不是普通的 Code Review——它发生在写代码**之前**，审查的是**计划本身**。

### 方法论框架：15 条认知模式

这是整个 gstack 中方法论密度最高的技能，它凝缩了大量工程管理经典著作的思维模型：

| # | 认知模式 | 出处 | 核心思想 |
|---|---------|------|---------|
| 1 | 状态诊断 | Larson《An Elegant Puzzle》 | 团队处于4种状态之一，干预方式完全不同 |
| 2 | 爆炸半径直觉 | — | 每个决策问"最坏情况影响多少系统/人" |
| 3 | **默认无聊** | McKinley《Choose Boring Technology》 | 公司只有3个创新代币，其他用已证实技术 |
| 4 | 增量优于革命 | Fowler | 绞杀者模式、金丝雀、重构而非重写 |
| 5 | 系统优于英雄 | — | 为凌晨3点疲惫的人设计 |
| 6 | 可逆性偏好 | — | 功能开关、A/B测试，降低犯错成本 |
| 7 | 失败是信息 | Allspaw, Google SRE | 无责任复盘、错误预算、混沌工程 |
| 8 | 组织=架构 | Conway定律, Skelton/Pais | 有意识地同时设计两者 |
| 9 | DX即产品质量 | — | 慢CI、差开发体验 → 差软件 |
| 10 | 本质vs偶然复杂性 | Brooks《No Silver Bullet》 | 在加东西前问：这解决真问题还是自造的 |
| 11 | 两周气味测试 | — | 工程师两周内交付不了小功能=架构问题 |
| 12 | 胶水工作意识 | Reilly《Staff Engineer's Path》 | 识别不可见的协调工作 |
| 13 | **先让改变变容易** | Kent Beck | 先重构，再实现，绝不同时进行 |
| 14 | 在生产中拥有代码 | Majors | 开发和运维之间没有墙 |
| 15 | 错误预算 | Google SRE | 99.9% SLO = 0.1% 用来发布的"预算" |

### 结构化审查流程：4轮 + Step 0

```
Step 0: 范围挑战（最重要，永远不跳过）
  ├── 现有代码是否已解决部分问题？
  ├── 最小变更集是什么？
  ├── 复杂性检查：>8文件 或 >2新类 = 代码味道
  ├── 搜索检查：框架有内置的吗？
  ├── TODOS 交叉引用
  ├── 完整性检查：AI 时代，完整版几乎免费
  └── 分发检查：产出物有构建/发布流水线吗？
       │
       ▼
  ┌─── 4轮审查（逐轮、逐问推进）────────────┐
  │ 1. 架构审查 → STOP 逐问确认              │
  │ 2. 代码质量审查 → STOP 逐问确认          │
  │ 3. 测试审查（含覆盖图表）→ STOP 逐问确认  │
  │ 4. 性能审查 → STOP 逐问确认              │
  └───────────────────────────────────────┘
       │
       ▼
  必需输出：
  ├── "不在范围内" 清单
  ├── "已有什么" 清单
  ├── TODOS.md 更新
  ├── ASCII 图表
  ├── 故障模式分析（关键缺口标记）
  └── 完成总结 + Lake Score
```

### 关键设计亮点

1. **"一个问题 = 一次提问"** — 拒绝合并问题，强制一个一个解决，避免遗漏
2. **Lake Score** — 衡量有多少推荐选择了"完整选项"而非捷径（AI 时代完整性几乎免费）
3. **逃生出口** — 没问题就跳过，显而易见的就直接做，只在有真正权衡时才提问
4. **未解决决策追踪** — 用户跳过的问题不会被静默默认，会在最后列出

## 二、`/investigate` — 系统化根因调试

### 核心方法论：**"Iron Law（铁律）— 没有根因调查就没有修复"**

这是一个**反直觉**的方法论——大多数人遇到 bug 的第一反应是"试着修一下"，而 Iron Law 说：**你修的不是 bug，你修的是症状。**

### 5 阶段调试流程

```
Phase 1: 根因调查          ← 收集证据，不形成假设
  │ 收集症状 → 阅读代码 → 检查 git 变更 → 复现
  │ 产出："根因假设: ..."
  │
  ▼
Phase 2: 模式分析          ← 匹配已知 bug 模式
  │ 6大模式：竞态条件、Nil传播、状态损坏、
  │         集成失败、配置漂移、过期缓存
  │
  ▼
Phase 3: 假设检验          ← 验证，不是猜测
  │ 添加临时日志 → 复现 → 证据匹配吗？
  │ ⚠️ 3次打击规则：3个假设都失败 → 停下！
  │
  ▼
Phase 4: 实现              ← 修根因，不修症状
  │ 最小差异 + 回归测试（无修复时失败，有修复时通过）
  │ ⚠️ >5个文件 → 爆炸半径警报
  │
  ▼
Phase 5: 验证与报告        ← 证明它修好了，不是"应该好了"
    结构化调试报告
```

### 关键设计亮点

1. **范围锁定** — 形成假设后，自动将编辑限制在受影响的模块，防止调试时"顺手"改了无关代码
2. **3 次打击规则** — 3 个假设失败后强制停下来，因为这可能不是 bug 而是**架构问题**
3. **红旗机制** — 检测危险行为模式：
   - "先临时修复" → 没有"临时"，修对或升级
   - 未追溯数据流就提修复 → 在猜测
   - 每次修复在别处暴露新问题 → 错误的层
4. **"永远不要说'这应该能修复它'"** — 运行测试，证明它

## 三、`/cso` — 首席安全官审计

### 核心方法论：**"像攻击者一样思考，像防御者一样报告"**

这是三个技能中**规模最大、最系统化**的，包含 15 个阶段（Phase 0-14），本质是一套完整的安全审计方法论。

### 15 阶段审计流水线

```
┌─────────────────────── 情报收集层 ───────────────────────┐
│ Phase 0:  架构心智模型 + 技术栈检测                        │
│ Phase 1:  攻击面普查（代码面 + 基础设施面）                 │
└──────────────────────────────────────────────────────────┘
           │
           ▼
┌─────────────────── 基础设施安全层 ────────────────────────┐
│ Phase 2:  秘密考古（git 历史泄露凭证）                      │
│ Phase 3:  依赖供应链（CVE + 安装脚本 + 锁文件）            │
│ Phase 4:  CI/CD 流水线安全                                │
│ Phase 5:  基础设施影子面（Docker / IaC）                   │
│ Phase 6:  Webhook & 集成审计                              │
└──────────────────────────────────────────────────────────┘
           │
           ▼
┌─────────────────── AI 安全层（独创）──────────────────────┐
│ Phase 7:  LLM & AI 安全（提示注入、输出净化、工具调用）      │
│ Phase 8:  AI 编码代理技能供应链                             │
└──────────────────────────────────────────────────────────┘
           │
           ▼
┌─────────────────── 应用安全层 ────────────────────────────┐
│ Phase 9:  OWASP Top 10 评估（A01-A10 逐类别）             │
│ Phase 10: STRIDE 威胁建模                                 │
│ Phase 11: 数据分类（RESTRICTED → PUBLIC 4级）              │
└──────────────────────────────────────────────────────────┘
           │
           ▼
┌─────────────────── 质量保证层 ────────────────────────────┐
│ Phase 12: 误报过滤 + 主动验证（22条硬排除 + 12条先例）      │
│ Phase 13: 发现报告 + 趋势追踪 + 修复路线图                 │
│ Phase 14: 保存报告（JSON 结构化）                          │
└──────────────────────────────────────────────────────────┘
```

### 关键设计亮点

1. **"零噪声 > 零遗漏"** — 每日模式要求 8/10 置信度才报告，宁可漏报也不误报。这是**反安全表演**的核心立场
2. **22 条硬排除规则 + 12 条先例** — 这是用大量误报经验沉淀出来的过滤器，比如：
   - React/Angular 默认 XSS 安全，只标记逃逸口
   - UUID 不可猜测，不标记
   - `docker-compose.yml` 本地开发用 root 不是发现
3. **AI 安全专项（Phase 7-8）** — 这是传统安全审计没有的，专门针对 LLM 时代的新攻击面：提示注入、RAG 投毒、工具调用劫持、成本攻击
4. **双模式设计**：
   - **每日模式** — 8/10 置信度门槛，零噪声
   - **全面模式** — 2/10 门槛，标记 `TENTATIVE`，用于月度深度扫描
5. **趋势追踪** — 跨审计报告比较（Resolved / Persistent / New），追踪安全态势方向

## 三角协作关系

三个角色不是孤立的，它们有明确的**审查链接**机制：

```
/plan-eng-review 完成后：
  ├── 检测到 UI 变更 → 建议 @plan-design-review
  ├── 重大产品变更 → 建议 @plan-ceo-review
  └── 安全敏感变更 → 建议 @cso          ←── 直接链接到 CSO
                                              │
/investigate 修复后：                          │
  ├── 同文件反复出 bug → 标记为架构味道        │
  └── 修复涉及 >5 文件 → 爆炸半径警报 → 可能触发 ──┘
```

## 方法论对比总结

| 维度 | plan-eng-review | investigate | cso |
|------|----------------|------------|-----|
| **角色** | 工程经理 | 调试专家 | 首席安全官 |
| **时机** | 编码前 | 出 bug 时 | 随时/定期 |
| **核心铁律** | 审查完才能写代码 | 没有根因不修复 | 零噪声 > 零遗漏 |
| **是否改代码** | ❌ 只出审查报告 | ✅ 修复 + 回归测试 | ❌ 只出安全报告 |
| **阶段数** | 4 轮 + Step 0 | 5 个 Phase | 15 个 Phase |
| **理论基础** | 15条认知模式（经典著作） | 6大 bug 模式 | OWASP + STRIDE |
| **逃生机制** | 无争议直接跳过 | 3次打击规则 | 22条硬排除 |
| **输出物** | 审查报告 + TODOS | 调试报告 | 安全态势报告(JSON) |
| **范围控制** | 复杂性检查(8文件/2类) | 范围锁定(Freeze) | 多模式(6种范围标记) |

**一句话总结**：开发阶段的方法论本质是 **"三道防线"** — 架构审查防止"造错了"，系统化调试防止"修错了"，安全审计防止"被攻破了"。
