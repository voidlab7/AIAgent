# Harness Engineering 应用到 AIAgent Pipeline 设计

> 来源：[Harness Engineering](../articles/Harness%20Engineering.md)
> 创建时间：2026-04-02
> 核心论点：LLM 在生产环境中执行复杂长周期任务时，Demo 惊艳但上线即崩——这不是模型能力问题，而是系统工程问题。

---

## 一、状态外置原则

**原文要点**：任务状态必须与模型上下文解耦，由 Harness 外部持久化。模型在每一步只被注入必要信息，而非依赖其记忆整个历史。

### 在 AIAgent 中的应用

1. **Pipeline 构建状态持久化**
   - 构建状态（成功/失败/进行中）必须写入外部存储（SQLite、文件系统、Git），而非仅存于 Agent 会话上下文
   - 每次构建触发时，从外部读取当前状态，而非假设 Agent 记得上次执行了什么
   - 进度信息通过 progress file 或数据库记录，Agent 每步只注入当前阶段所需的上下文

2. **跨会话任务恢复**
   - 长任务（如多步骤构建、批量内容发布）中断后，新的 Agent 会话能从外部状态文件恢复
   - 避免因会话过期导致任务丢失，必须重新开始

3. **实践检查清单**
   - [ ] 任务状态是否存储在 Agent 上下文之外？
   - [ ] 新会话是否能从外部状态恢复中断的任务？
   - [ ] 每步执行时是否只注入必要的上下文信息？

## 二、工具小粒度 + 高确定性

**原文要点**：工具接口要尽可能简单（小粒度、高确定性）。大而全的工具接口会增加模型出错概率。

### 在 AIAgent 中的应用

1. **MCP 工具设计原则**
   - 每个工具只做一件事，参数尽量少，返回值结构明确
   - 避免"瑞士军刀"式工具——把多个操作合并到一个工具中
   - 工具必须**幂等**：相同输入多次调用产生相同结果，不会产生副作用

2. **蓝盾 Pipeline MCP 工具优化**
   - `v4_stream_user_openapi_trigger`：触发构建，参数明确（projectId、pipelineId、path、branch）
   - `v4_user_build_status`：查询构建状态，输入构建 ID，输出结构化状态信息
   - `v4_user_build_list`：查询构建历史，支持分页
   - 每个工具职责单一，可独立测试

3. **小红书 MCP 工具优化**
   - 发布、搜索、点赞、评论等操作各自独立工具
   - 避免将"搜索+分析+发布"合并为一个复合工具
   - 每个工具的错误码明确，便于系统级错误分类

4. **实践检查清单**
   - [ ] 每个工具是否只做一件事？
   - [ ] 工具是否幂等（重复调用不产生副作用）？
   - [ ] 工具返回值是否结构化、可解析？
   - [ ] 错误码是否明确分类？

## 三、系统级 QA：自动化验证比模型自评更可靠

**原文要点**：不能仅靠模型自行判断任务是否完成，需要引入独立的验证机制（单元测试、UI 自动化、第二模型评审）。

### 在 AIAgent 中的应用

1. **Pipeline 构建验证**
   - 构建完成后，通过 `v4_user_build_status` 查询实际状态，而非让 Agent "认为" 构建成功了
   - 部署后通过 HTTP 健康检查验证服务可用性，而非假设部署一定成功

2. **小红书发布验证**
   - 发布内容后，通过 `get_feed_detail` 验证内容确实已上线
   - 不能仅凭 API 返回 200 就认为发布成功，需要二次确认

3. **代码变更验证**
   - 代码修改后运行 linter 和测试，而非依赖 Agent 自检
   - `read_lints` 工具提供了系统级验证能力

4. **实践检查清单**
   - [ ] 关键操作后是否有独立的验证步骤？
   - [ ] 验证机制是否依赖外部系统而非 Agent 自评？
   - [ ] 验证失败时是否有明确的恢复策略？

## 四、Context Reset 策略

**原文要点**：相比在同一会话内压缩历史（Context Compaction），更推荐 Context Reset + 结构化 Handoff——清空历史会话，通过外部可验证的结构化文档传递状态。

### 在 AIAgent 中的应用

1. **长任务中断恢复**
   - 当 Agent 会话因 token 限制或超时中断时，不尝试压缩上下文继续
   - 而是将当前状态写入结构化文档（如 `handoff.json`、progress file）
   - 新会话启动时从结构化文档恢复状态，重新构建上下文

2. **自动化任务的 Handoff 设计**
   ```
   handoff.json 示例：
   {
     "task": "批量发布小红书内容",
     "phase": "publishing",
     "completed": ["选题1", "选题2"],
     "current": "选题3",
     "pending": ["选题4", "选题5"],
     "results": {
       "选题1": {"feed_id": "xxx", "status": "published"},
       "选题2": {"feed_id": "yyy", "status": "published"}
     },
     "error_log": []
   }
   ```

3. **vs Context Compaction**
   | 方案 | 优势 | 劣势 |
   |------|------|------|
   | Context Compaction | 会话连续，无需外部文件 | 压缩会丢失细节，错误信息累积污染 |
   | Context Reset + Handoff | 干净的上下文，无历史包袱 | 需要设计良好的 Handoff 文档结构 |

4. **实践检查清单**
   - [ ] 长任务是否有阶段性的状态快照？
   - [ ] 状态快照是否结构化（JSON/Markdown），可被新会话解析？
   - [ ] 新会话能否从快照无缝恢复，无需人工介入？

## 五、错误分类与恢复策略

**原文要点**：引入系统级错误分类（临时性、可回滚、权限、逻辑错误）和对应策略（重试、回滚、人工介入）。

### 错误分类框架

| 错误类型 | 示例 | 恢复策略 |
|----------|------|----------|
| **临时性错误** | 网络超时、API 限流 | 自动重试（指数退避） |
| **可回滚错误** | 构建配置错误、代码语法错误 | 回滚到上一个检查点，修复后重试 |
| **权限错误** | SSH 密钥失效、Token 过期 | 暂停任务，通知人工介入 |
| **逻辑错误** | 业务规则违反、数据校验失败 | 记录错误，跳过或人工决策 |

### 在 AIAgent 中的应用

- Pipeline 构建失败 → 判断是临时性（重试）还是逻辑错误（通知人工）
- SCP 部署失败 → 判断是网络问题（重试）还是权限问题（通知人工）
- 小红书发布失败 → 判断是限流（等待后重试）还是内容违规（跳过并记录）

## 六、Human-in-the-Loop 设计

**原文要点**：在关键决策点引入人工确认节点，而非让 Agent 全自动执行所有操作。

### 在 AIAgent 中的应用

- **破坏性操作前确认**：`/careful` 模式——删除文件、force push 等操作前警告
- **部署前确认**：构建完成后、部署前，展示变更摘要请求确认
- **异常情况升级**：超过 3 次自动重试后，停止并通知人工

## 七、构建路线图（适用于 AIAgent）

```
Phase 1 - 最小可用 Harness（已完成）
├── Environment：本地开发环境 + SSH 远程环境
├── Tools：MCP 工具（蓝盾 Pipeline、小红书、浏览器）
└── Simple Loop：单次命令触发单次执行

Phase 2 - 增加控制层（部分完成）
├── Control：/careful 安全模式、/freeze 编辑限制
├── Memory：.context/ 目录存储任务状态、retros 历史
└── 基本错误处理：工具调用失败时的重试逻辑

Phase 3 - 完整 Task Loop（待建设）
├── 任务分解与阶段管理
├── Checkpoint 与断点续跑
├── 系统级 QA 验证
├── 结构化 Handoff 文档
└── 多 Agent 角色接力

Phase 4 - 高级能力（远期）
├── 自动化定时任务（已支持 automation_update）
├── 团队协作模式（已支持 team_create）
└── 自我评估与优化
```
