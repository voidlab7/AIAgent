本文系统性地介绍了Harness Engineering（ harness 工程）这一新兴概念，旨在解决大语言模型（LLM）在真实生产环境中执行复杂长周期任务时面临的稳定性、可控性和可审计性挑战。文章首先阐述了为何在模型能力强大的今天，工程系统（Harness）成为决定Agent能否真正完成工作的关键，然后深入剖析了Harness的核心定义、设计哲学、架构组件，并辅以大量实证案例和最佳实践，最终为从零开始构建Harness提供了清晰的路线图。

文章重点总结

背景与核心问题：大语言模型能生成看似完美的答案，但在生产环境中独自运行复杂长任务时，普遍存在Demo惊艳但上线即崩、跨天长任务无法延续、代码库被写碎、审计回溯困难等问题。这些问题本质上是系统工程问题，而非模型能力问题，因此需要Harness（模型外部的控制系统）来解决。

Harness的定义与核心概念：

定义：Harness是包裹在模型外部，使其能在真实环境中长期、安全、可控执行任务的系统层，即围绕LLM/Agent构建的确定性控制系统。

核心组件：一个生产级Harness如同一个小型操作系统，包含五大核心子系统：

Environment (环境)：提供受控的工作世界（文件系统、沙箱终端、浏览器等）。

Tools (工具系统)：对环境能力进行抽象封装，提供简单、可审计的Action。

Control (控制层)：管理执行边界（步数/时长限制、状态机、错误处理、权限控制）。

Memory (记忆/状态系统)：解决长任务/跨会话的状态持久化问题（进度文件、架构文档、Git历史）。

Evaluation (自动化检查/验证)：让系统自我评估结果（单元测试、UI自动化、第二模型评审）。

设计哲学：采用确定性外壳 (Deterministic Shell) 包裹非确定性内核 (Stochastic Core) 的模式。凡是能用确定性机制表达的约束（如权限、状态机、回滚），就应写入系统代码而非仅依赖Prompt。

关键机制与工程实践：

Task Loop (任务循环)：这是Harness的核心体现，将无状态的Agent Loop进化为任务级管理系统。它包含五大核心机制：任务分解与阶段管理、检查点(Checkpoint)与断点续跑、错误处理与自动恢复、资源管控与终止条件、人机协作节点(Human-in-the-Loop)。

状态管理：任务状态必须与模型上下文解耦，由Harness外部持久化（如progress file），模型在每一步只被注入必要信息，而非依赖其记忆整个历史。

错误恢复：引入系统级错误分类（临时性、可回滚、权限、逻辑错误）和策略（重试、回滚、人工介入），而非将所有错误抛回给模型处理。

Context策略：相比在同一会话内压缩历史(Context Compaction)，更推荐Context Reset + 结构化Handoff，即清空历史会话，通过外部可验证的结构化文档（如handoff.json）传递状态，更能避免错误信息的污染。

实证案例与效果：文章列举了多个案例证明Harness的巨大价值：

性能提升：仅优化Harness而不更换模型，就使编程基准通过率从42%提升至78%。

生产实践：Stripe的“Minions”系统每周处理1300个无人值守PR；Cursor的“Self-Driving Codebases”峰值达每小时1000个commit；OpenAI Codex项目在5个月内由Agent编写约100万行代码，人类未写一行。

质量对比：在构建2D游戏编辑器的任务中，使用三Agent Harness的方案虽然耗时更长、成本更高，但产出了功能丰富且真正可玩的产品，远胜单Agent方案表面光鲜但实际不可用的输出。

反模式与原则：

常见陷阱：把所有规则写进Prompt、只追求多Agent忽略Task Loop、长期状态只放上下文、没有系统级QA仅靠模型自评、没有幂等性设计。

五大工程原则：

尽量减少模型需要记住的内容（状态外置）。

把规则写进系统，而不是写在Prompt里（系统强制优于语言提示）。

工具接口要尽可能简单（小粒度、高确定性）。

任务状态必须持久化（支持中断恢复）。

系统必须可观测、可诊断（记录完整执行轨迹）。

构建路线与未来展望：文章给出了从最小可用Harness（环境+工具+简单Loop）开始，逐步迭代增加Control/Memory/Evaluation，再到引入多Agent角色接力的实践路线。未来，Harness将向标准化、模板化发展，与工作流引擎融合，但其核心的“控制论”价值（提供约束、反馈、安全）不会因模型变强而消失，只会演变。
