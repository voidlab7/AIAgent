## 🎯 一句话总结

终端 Agent 不是万能的，用好它的关键是分清「哪 3 类活该给」「哪 3 类活别给」，核心原则是「人定边界，Agent 跑第一轮，人做最终判断」。

---

## 💡 核心知识点

### #1 Agent 最适合的 3 类任务

1. **边界清晰的实现任务** — issue 写清楚、改动明确、有测试可验
2. **重复出现的工作流任务** — 文档/配置/测试补齐、脚手架、PR 跟进
3. **先跑第一轮的探索任务** — 扫代码、列改动点、写初版方案，人再拍板

### #2 Agent 不该接的 3 类任务

1. **需求自己没定清** — 模糊交给 Agent 只会放大模糊
2. **高风险真实动作** — 生产环境、账号权限、资金操作，必须人工接管
3. **强依赖视觉和业务感觉** — 复杂 UI 微调、产品取舍

### #3 下任务前的 4 问框架

1. 目标能不能一句话说清？
2. 验收标准是不是客观的？（测试通过/截图对齐/build 成功）
3. 出错成本高不高？
4. 返工时谁来收尾？

### #4 风险分级规则

- 低风险 → Agent 多跑
- 中风险 → Agent 提方案，人确认
- 高风险 → Agent 只做准备，人最终执行

---

## 🔧 Agent 任务分配三步法

```
Step 1：人定边界（目标清晰 + 验收标准）
Step 2：Agent 跑第一轮（探索 + 初版实现）
Step 3：人做最终判断（Review + 决策 + Merge）
```

## ✅ 行动建议

1. 下次用 AI 前，先过「4 问框架」，拒绝把模糊需求直接丢给 AI
2. 对照「3 类该给 / 3 类不该给」，盘点 AI 编程习惯
3. `plan-eng-review` Rule 就是「人定边界」的具体落地，每次 feat 前先跑
