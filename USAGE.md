## Rules 详解（29 个）

### 🔧 开发阶段

---

#### `/plan-eng-review` — 工程方案审查

**触发词**：`"审查架构"` `"工程审查"` `"plan review"` `"engineering review"` `"有计划想开始写代码了"`

**做什么**：在写代码之前，逐问题锁定执行计划。输出架构图、数据流、边界情况、测试覆盖、性能考虑。

**最适合**：任何需要修改 3+ 个文件的功能，先跑这个，再动手写代码。

**例子**：
```
你：给 PawMBTI 加邀请码系统，帮我做工程审查
→ AI 输出架构方案、数据模型、API 设计、测试计划，逐问确认
```

---

#### `/review` — PR 预合并审查

**触发词**：`"review"` `"审查这个 PR"` `"代码审查"` `"check my diff"`

**做什么**：分析当前分支 diff，检测 SQL 注入、竞态条件、枚举完整性等结构性问题。机械问题直接修，需要判断的才问你。

**最适合**：提交前的最后一道关卡。

---

#### `/investigate` — 系统化根因调试

**触发词**：`"investigate"` `"为什么坏了"` `"debug"` `"根因分析"` `"这个错误怎么回事"`

**做什么**：Iron Law 调试法——先找根因再修复。绝不猜测，通过代码追踪、假设检验找到真正的根因。

**最适合**：Bug 出现时第一个调用。

**铁律**：3 个假设都失败就停下来，可能是架构问题不是 Bug。

---

#### `/cso` — 安全审计

**触发词**：`"cso"` `"安全审计"` `"security review"`

**做什么**：15 阶段安全审计流水线。OWASP Top 10、STRIDE 威胁建模、依赖供应链、AI 安全专项（提示注入等）。

**最适合**：上线前、有敏感功能时。

---

### 🚀 发布阶段

---

#### `/ship` — 一键发布工作流

**触发词**：`"ship"` `"发布"` `"推到 main"` `"创建 PR"` `"代码准备好了"`

**做什么**：全自动流程——检测基础分支 → 运行测试 → 审查 diff → 更新版本号 → 生成 CHANGELOG → 提交 → 推送 → 创建 PR。

**最适合**：日常功能开发完成后一键发布。

**注意**：它会自动做所有事，只在测试失败或有判断题时停下来。

---

#### `/land-and-deploy` — 合并 + 部署 + 验证

**触发词**：`"land-and-deploy"` `"合并并部署"` `"merge and deploy"`

**做什么**：合并 PR → 等待部署完成 → 验证生产环境。检测项目类型，根据变更范围动态调整验证深度。

**最适合**：PR 创建后，在服务器上确认部署成功。

---

#### `/document-release` — 更新文档

**触发词**：`"document-release"` `"更新文档"` `"文档同步"`

**做什么**：扫描所有已变更的功能，更新 README、API 文档、CHANGELOG，确保文档与代码同步。

---

#### `/setup-deploy` — 配置部署设置

**触发词**：`"setup-deploy"` `"配置部署"` `"how do I deploy"`

**做什么**：检测部署平台（Fly.io、Vercel、Render 等），配置健康检查端点，持久化到项目文档。

## 注意事项

1. **需要审批的操作**：`git push`、`scp`、`ssh` 到服务器 — 这些会弹出确认框
2. **发布前必须确认**：小红书发布、GitHub Release 创建 — 不会自动触发
3. **安全例外**：`rm -rf node_modules/dist/build` 不需要确认
4. **服务器密码**：存在环境变量 `$SERVER_PASSWORD`，不会明文出现在命令里
5. **小红书 MCP 前置**：使用小红书相关功能前，确保 `xiaohongshu-mcp` 服务在 18060 端口运行

## 每次下任务的 SOP

> 基于《终端 Agent 只该接 3 类活》的方法论，结合本项目工具链落地。
> 核心原则：**人定边界 → Agent 跑第一轮 → 人做最终判断**

### 下任务前过 4 问

在说任何任务之前，先在脑子里确认：

| 问题 | 通过标准 | 没过怎么办 |
|------|---------|-----------|
| 目标能一句话说清吗？ | 说得出「改什么文件、加什么功能、完成状态是什么」 | 先想清楚再说 |
| 验收标准客观吗？ | 能用「点击有效 / 页面正常 / 不报错」验证 | 补充验收标准 |
| 出错成本高吗？ | 判断改动范围（文件数 / 是否影响生产） | 先 `careful`，高风险只让 AI 做准备 |
| 返工时谁来收尾？ | 做歪了我能快速发现并修回来 | 提前想好回滚方案 |

### 好的下任务句式

```
❌ 坏的句式（只有目标）：
"给 pawmbti 加个邀请码功能"

✅ 好的句式（目标 + 验收）：
"给 QuickLandingPage 的「开始测试」按钮加邀请码验证——
 点击时如果 localStorage 没有 _ml_invite_ok 就弹 InviteModal，
 有的话直接跳转测试页。
 验收：清 localStorage 后点击弹窗出现，有 key 后不弹直接跳转。"
```

**一句话记住**：「改什么」+「验收是什么」，缺一不可。

### 按任务复杂度选路径

```
复杂任务（改 3+ 文件 / 新功能 / 不确定边界）：
  → plan-eng-review → 确认方案 → 实现 → review → ship

简单任务（改 1-2 文件 / 自己清楚边界）：
  → 用好句式直接下任务 → review → ship

探索任务（不确定怎么做 / 需要先摸底）：
  → 说「先帮我分析 XX，不要改代码」→ 看 AI 给的分析 → 再决定方案
```

### 每次写代码的固定流程

```
1. 过 4 问（1 分钟）
      ↓
2. 复杂 → plan-eng-review   简单 → 好句式直接下
      ↓
3. AI 实现
      ↓
4. review（不要跳过，几乎不花时间）
      ↓
5. ship（会自动跑测试、提交、创建 PR）
      ↓
6. 部署 → pawmbti-release → scp-upload（需手动审批）
```

### 3 类不该给 AI 的任务

| 类型 | 例子 | 正确做法 |
|------|------|---------|
| 需求没定清 | "优化一下用户体验" | 先自己想清楚要改什么 |
| 高风险生产操作 | 直接 push main / 删数据库 | `careful` 模式 + 手动审批 |
| 强依赖视觉判断 | "这个按钮感觉不好看，调一下" | 自己先定好目标状态再下任务 |
